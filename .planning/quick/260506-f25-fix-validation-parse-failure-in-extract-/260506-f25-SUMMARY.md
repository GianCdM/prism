# Quick Task 260506-f25: Fix Validation Parse Failure in Extract Summary

Restored Prism's extraction pipeline by hardening `_parse_validation_output` to handle Sonnet's actual multi-block JSON output and aligning the validator prompt to request a canonical single JSON array.

## Context

Symptom: `prism extract` runs completed with `approved=0, rejected=0, modified=0` regardless of how many candidates Sonnet had reviewed. No engram files were ever materialised even when Sonnet had clearly approved candidates.

Root cause: Sonnet was emitting one fenced ```json block per candidate (matching the validator.md instruction "For each candidate, output your decision as a JSON object"). `_parse_validation_output` used `re.search` (single match) and then expected a list, so it received a single dict, failed `isinstance(decisions, list)`, and the tally loop never executed. The outer fallback then counted files in `engrams_dir`, but Sonnet hadn't moved any candidates into engrams yet, so the count remained zero.

## Changes

### `lib/extract.py` — `_parse_validation_output` (Task 1, commit `dccd1a4`)

Before:
- `re.search(r"\`\`\`json\s*\n(.*?)\n\s*\`\`\`", output, re.DOTALL)` — matched only the first block
- Treated parsed result as a list; if it was a dict, the `isinstance` check failed silently and tally was skipped
- Decisions list ended up empty in the multi-block case

After:
- `re.findall(...)` collects ALL fenced ```json blocks
- Three explicit branches:
  - **multi-block (`len(matches) > 1`)**: parse each block independently, append dicts and extend lists into a single decisions list, skip blocks that fail to parse
  - **single-block (`len(matches) == 1`)**: parse the one block; if list use as-is (back-compat), if dict wrap into 1-element list
  - **no-block (`len(matches) == 0`)**: parse whole body, wrap dict into list as needed (existing fallback path)
- New `parsed_any` guard trips the outer `JSONDecodeError` fallback when every individual block fails to parse, so the file-count recovery still kicks in for total parse failure
- Tally loop now defensively skips non-dict entries
- Function signature, return shape, and `_log_validation` call unchanged
- `_apply_validation_results` not touched

### `agents/validator.md` and `/Users/lara.baseggio/.prism/agents/validator.md` (Task 2, commit `ab418b1`)

`## Output` section rewritten:
- "For each candidate, output your decision as a JSON object:" → "Output your decisions as a single JSON array. Each element is one decision object covering one candidate:"
- Example block changed from a single object `{...}` to an array `[ {...} ]` wrapping the same object shape
- All five gates (constitution, evidence, contradiction, safety, novelty), decision rules, and Rules section unchanged
- Repo source and installed copy at `~/.prism/agents/validator.md` are byte-identical in the changed section
