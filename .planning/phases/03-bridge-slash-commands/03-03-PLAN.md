The SKILL.md must instruct Claude to perform these steps when invoked:

**1. Configuration Resolution:**
- Read registry URL from `~/.prism/config.json` key `registry_url`. If `--registry NAME` flag provided, look up that registry's URL (Phase 3: just use the single `registry_url` from config; Phase 4 adds multi-registry via `registries.json`).
- Read auth token from `REGISTRY_TOKEN` environment variable.
- If registry URL is empty: print "No registry configured. Run: prism config registry_url <URL>" and stop.
- If no token: print "Set REGISTRY_TOKEN environment variable with your API token" and stop.

**2. Skill Discovery:**
- Scan `_analysis/extracted_skills_codebase/` for skill directories (each must contain `plugin.json` + `SKILL.md`).
- Validate each skill's plugin.json has all required fields per schema: name, description, author, repository, category, source, commit_date, source_hash.
- Report validation failures and skip invalid skills.

**3. Delta Tracking (SKILL-10):**
- Read `_analysis/.published.json` if it exists (create empty `{}` if not).
- For each valid skill, compute content hash: SHA256 of `plugin.json` bytes + `SKILL.md` bytes concatenated, take first 12 hex chars.
- Compare against `.published.json` entry for the target registry (use `"default"` as registry name in Phase 3).
- If `--all` flag: skip delta check, publish everything.
- Otherwise: only include skills where hash differs or no entry exists.

**4. Publishing:**
- Build request payload: JSON array of skill objects, each with plugin.json fields plus `content` field containing SKILL.md text (not base64 -- simpler than Lens approach).
- POST to `{registry_url}/api/skills/publish` with `Authorization: Bearer {token}` header.
- Handle response: report success count, any errors.

**5. Update Delta Tracking:**
- After successful publish, update `_analysis/.published.json` with new content hashes and timestamps for published skills.
- Write atomically (write to .tmp then rename).

**6. Summary:**
- Print count of skills published, skipped (unchanged), and failed.
- If any published: "Published {N} skills to {registry_url}"
- If all unchanged: "All skills up to date. Use --all to force republish."

**Flags the command must support:**
- `--registry NAME` -- target a specific registry (Phase 3: informational, uses config registry_url)
- `--all` -- publish all skills regardless of delta state

**Important differences from Lens:**
- NO GitHub publishing path (Worker-only per PROJECT.md decision)
- Registry URL from `~/.prism/config.json`, NOT from `REGISTRY_API_URL` env var
- Auth token still from `REGISTRY_TOKEN` env var (per research recommendation)
- Delta tracking via `.published.json` (Lens had no delta tracking)
- Simpler: one target, one API, deterministic behavior

Write the SKILL.md as a Claude Code slash command -- it should be comprehensive instructions that Claude follows when the user invokes `/publish-skills`. Include all the logic above as step-by-step instructions Claude executes using bash/python tools.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Registry API -> local display | Remote registry content displayed to user; could contain malicious instructions in skill descriptions |
| Auth token -> HTTP request | REGISTRY_TOKEN sent as Bearer token to configured registry URL |
| Local config -> API target | registry_url from config.json determines which server receives publish requests |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07 | S (Spoofing) | publish-skills auth | mitigate | Bearer token auth via REGISTRY_TOKEN env var; HTTPS enforced by Worker |
| T-03-08 | I (Info Disclosure) | publish-skills payload | accept | Skill content is intended to be shared; publishing is an intentional action |
| T-03-09 | T (Tampering) | advise-skills registry content | accept | Registry is controlled by team; Worker validates on publish. User trusts configured registry URL. |
| T-03-10 | I (Info Disclosure) | REGISTRY_TOKEN in env | mitigate | Token stored in env var (not config.json) to avoid accidental commit; slash commands reference env var only |
</threat_model>
